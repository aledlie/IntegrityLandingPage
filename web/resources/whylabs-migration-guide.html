What Happened to WhyLabs?

In late 2024, WhyLabs—a Seattle-based startup that helped define the AI Observability category—announced it was discontinuing operations. Reports confirmed the shutdown was the result of a quiet acquisition by Apple.

In response to their shutdown, WhyLabs took commendable steps to help their users:

  • Open-sourced the platform under Apache 2 license (January 23, 2025)
  • Extended SaaS access until March 9, 2025
  • Continued maintaining whylogs and LangKit as open-source projects

Your Migration Options

You have three main paths forward:

Option 1: Self-Host WhyLabs Open Source

WhyLabs has released their platform as open source. You can run it on your own infrastructure.

Pros:

  • Familiar interface and features
  • No immediate code changes required
  • Free (Apache 2 license)

Cons:

  • Requires DevOps resources to deploy and maintain
  • No official support (community only, no contributions accepted)
  • Highcharts license required for visualizations (additional cost)
  • Uncertain long-term maintenance
Licensing Note

The open-source WhyLabs platform requires a separate Highcharts license for data visualizations. This can cost $400-$6,600+ depending on your use case (internal tools vs SaaS deployment).

Option 2: Migrate to Another Platform

Several AI observability platforms can replace WhyLabs functionality:

Platform Best For WhyLabs Feature Coverage
Integrity Studio LLM monitoring, EU AI Act compliance 85%+ with enhanced compliance
Arize AI Large enterprise ML teams 90%+ (complex setup)*
Fiddler AI ML monitoring + LLM guardrails 85-90%
Langfuse LLM-only use cases (fully open source) 65% (LLM focused)*

* Arize AI also offers Phoenix, an open-source observability tool if you prefer to self-host.
* Langfuse open-sourced all formerly commercial features (evaluations, experiments, playground) under MIT license in June 2025.

Option 3: Build Custom Solution with whylogs

WhyLabs' core logging library, whylogs, remains actively maintained as open source. You can continue using it with custom dashboarding.

Pros:

  • Minimal code changes if you're already using whylogs
  • Privacy-preserving data profiles still work
  • Flexible—use any visualization/storage backend

Cons:

  • No built-in dashboards or alerting
  • Significant development effort required
  • Must build or buy monitoring UI
  • Community-maintained: security patches may be slower than commercial support

Step-by-Step Migration Guide

Here's the recommended migration process, designed to minimize disruption:

  • Set up your organization and team members

3
Install SDK and Update Integration Code
~2-4 hours

Replace WhyLabs SDK calls with Integrity Studio equivalents:

  • Install the Integrity Studio SDK
  • Update import statements
  • Migrate logging calls (see code examples below)
  • Configure environment variables
4
Recreate Monitors and Alerts
~1-2 hours

Set up equivalent monitoring in Integrity Studio:

  • Configure drift monitors with your thresholds
  • Set up data quality checks
  • Create alert rules and notification channels
  • Configure LLM-specific guardrails
5
Run Parallel Monitoring
~1-2 weeks

Validate your migration before fully switching:

  • Send data to both platforms simultaneously
  • Compare drift detection results
  • Verify alert thresholds are calibrated correctly
  • Train team on new dashboard and workflows
Parallel Monitoring Security
  • Minimize overlap period to 1-2 weeks maximum to reduce dual API key exposure
  • Use separate credential stores for WhyLabs and Integrity Studio API keys
  • Implement feature flags to cleanly switch between platforms rather than running both indefinitely
  • Rotate credentials after migration - generate new Integrity Studio API keys after removing WhyLabs
6
Complete Migration
~1 hour

Finalize the transition:

  • Remove WhyLabs SDK and configuration
  • Update documentation and runbooks
  • Verify all alerts and monitors are active
  • Archive exported WhyLabs data

Handling Personal Identifiable Information (PII)

When logging user-related data for compliance or monitoring:

  • Always hash user identifiers using SHA-256 with a consistent salt
  • Never log directly: email addresses, names, IP addresses, or session tokens
  • Consider GDPR/privacy requirements: Even hashed IDs may be personal data under some regulations
  • Use consistent hashing: Same user should produce same hash for tracking across sessions
Secure User ID Hashing Helper Python 
import hashlib
import os

# Generate a salt once and store securely (environment variable or secrets manager)
# This salt should be consistent across your application for user tracking
USER_ID_SALT = os.environ["USER_ID_SALT"]  # Random 32+ character string

def hash_user_id(user_id: str) -> str:
    """Create a one-way hash of user ID for privacy-preserving logging.

    Uses SHA-256 with salt to prevent rainbow table attacks.
    Truncated to 16 chars for practicality while maintaining uniqueness.
    """
    if not user_id:
        return "anonymous"
    salted = f"{USER_ID_SALT}{user_id}".encode('utf-8')
    return hashlib.sha256(salted).hexdigest()[:16]

# Usage
hashed_id = hash_user_id("user@example.com")  # Returns: "a3f2b8c1e9d4f7a2"

Error Handling Best Practices

Always implement proper error handling to prevent monitoring failures from affecting your application:

Resilient Logging Pattern Python 
import os
import logging
from integrity_studio import IntegrityClient, IntegrityAPIError

logger = logging.getLogger(__name__)
client = IntegrityClient(api_key=os.environ["INTEGRITY_API_KEY"])

def log_interaction_safely(prompt: str, response: str, **kwargs):
    """Log LLM interaction without blocking the main application."""
    try:
        client.log_llm_interaction(
            prompt=prompt,
            response=response,
            **kwargs
        )
    except IntegrityAPIError as e:
        # Log error but don't crash the application
        logger.error(f"Failed to log to Integrity Studio: {e}")
    except Exception as e:
        # Catch unexpected errors to prevent monitoring from breaking production
        logger.exception(f"Unexpected error in monitoring: {e}")

Exporting Your WhyLabs Data

Before WhyLabs access ends on March 9, 2025, export your data:

What to Export

  • Data Profiles: Historical profile data for trend analysis
  • Alert Configurations: Your threshold and notification settings
  • Custom Metrics: Any custom metric definitions
  • Dashboard Configurations: Layout and visualization preferences
  • API Keys and Org Settings: For reference during migration
Data Export Security
  • Never hardcode API keys - use environment variables or a secrets manager
  • Encrypt exported data - profiles may contain sensitive model behavior patterns
  • Secure file permissions - restrict access to exported files (chmod 600)
  • Delete credentials after migration - rotate or revoke WhyLabs API keys when done
  • Audit storage location - use access-controlled storage with logging

Export Using WhyLabs API

API Endpoint Note

The WhyLabs API is primarily designed for data ingestion. Bulk export endpoints may have limited availability as the platform winds down. We recommend: (1) contacting WhyLabs support directly for export assistance, (2) checking if you retained local profile copies before upload (whylogs can write to disk), or (3) using the whylabs-toolkit package which may offer additional export capabilities. Test API access before relying on this script.

export_whylabs_data.py Python 
import os
import requests
import json
from datetime import datetime, timedelta

# Load credentials from environment variables
WHYLABS_API_KEY = os.environ["WHYLABS_API_KEY"]
ORG_ID = os.environ["WHYLABS_ORG_ID"]
DATASET_ID = os.environ.get("WHYLABS_DATASET_ID", "model-1")

headers = {
    "X-API-Key": WHYLABS_API_KEY,
    "Content-Type": "application/json"
}

# Export profile data for last 90 days
end_time = datetime.utcnow()
start_time = end_time - timedelta(days=90)

response = requests.get(
    f"https://api.whylabsapp.com/v0/organizations/{ORG_ID}"
    f"/models/{DATASET_ID}/profiles",
    headers=headers,
    params={
        "startTime": start_time.isoformat(),
        "endTime": end_time.isoformat()
    }
)

# Save to file with restricted permissions
output_file = "whylabs_export.json"
with open(output_file, "w") as f:
    json.dump(response.json(), f, indent=2)
os.chmod(output_file, 0o600)  # Restrict to owner only

print(f"Exported {len(response.json())} profiles to {output_file}")
print("Remember to encrypt this file before long-term storage!")

Frequently Asked Questions

Can I keep using whylogs with Integrity Studio?

Yes! whylogs remains open source and actively maintained. Integrity Studio can ingest whylogs profiles directly, so you can keep your existing data logging code and only change where profiles are sent.

What about LangKit?

LangKit also remains open source. However, Integrity Studio includes equivalent (and enhanced) LLM metrics built-in, so you may not need LangKit at all. If you prefer LangKit's specific metrics, you can use both together.
